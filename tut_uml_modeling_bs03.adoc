== モデルとコードの対応づけ

[.lead]
プログラムがどのように動作するのか検討するときには、振る舞いのモデルを使います。+
振る舞いのモデルを作成する前に、構造のモデルと振る舞いのモデルがどのようなコードとして実現されるのかについて、このチュートリアルの開発で用いる方式を決めておきましょう。 +
このことを「モデルとコードの対応づけ」または「実装方式の設計」と呼ぶことにします。

TIP: このような対応づけを「モデル変換ルール」と呼ぶこともあります。また、開発プロセスのなかでこの対応づけを検討する工程を「方式設計」と呼ぶ人もいます。


.この章の役割についてもう少し
[sidebar]
--
この章で検討したいことは、ボウリングのスコアに関してどのようなモデルを作るのか、という話題から少し離れています。
モデルをどのように実装するのかということは、モデルの表し方とコードの表し方の間の関係を定めるものであって、ボウリングのスコアをどのようなモデルとして表すかということとは独立している話題だからです。

もし、この章の役割がピンとこないとか、実装についての議論にはあまり関心がないのであれば、最初はこの章を読み飛ばしてみてもかまいません。
後の章へ進めば、実装する人がモデルを使う際には、モデルとコードを対応づける必要があり、事前に実装方式を決めておかなければ、実装者が実装方法について考える（実装時に「どのように作るのか」を設計する）必要があることに気づくでしょう。
そのときは、この章に戻って、どうして実装方式の設計が必要なのか考えてもらえたらと思います。
--

=== なぜ対応づけの検討が必要なのか

ソフトウェア開発でよく目にする事態として、設計と実装が乖離してしまい、頼れるのは動いているコードだけというものがあります。
このとき、多く人が「設計書がないから」「設計書が更新できていないから」と説明します。
このことについて、もう少し詳しく考えてみましょう。

ソフトウェアの開発に使う設計書には次のような2つの役割があります。

.設計書の役割
* *「何を作るか」* について、構造と振る舞いをもって示す。
* *「どのようにして作るか」* について、設計書と実装の対応づけをもって示す。

「何を作るか」の設計書は、どんな機能を提供し、それがどのような動作・振る舞いとなっているかを示す設計書です。
例えば、「機能的な要求の発見・獲得→要求の分析→外部設計→内部設計」という開発プロセスがあるとき、「要求分析のユースケース図」、「外部設計のクラス図・シーケンス図」、「内部設計のクラス図・ステートマシン図」などを作成します。
これらが「何を作るか」を示すための設計書です。

では、「どのようにして作るか」の設計書とはどんなものでしょうか。
これは、例えば、設計書にクラスがあったらC言語やRubyの構造体に対応づけるとか、状態遷移は2次元配列を使って遷移表として実装しようとか…そういった、設計書に記載した要素に対して、実装するときにコードのどの部分にどのように反映するかを示した設計書です。

先に挙げた「何を作るか」の設計書には、「どのような実装方法で実装するのか」は書いてありません。
しかし、「どのようにして作るか」の設計書がないと、その側面は実装者がコードを作成しながら考えることになります。
そうなると、「どのようにして作るか」は、実装者ごとに考えることになり、また、同じ実装社でも実装する毎に異なってきます。しかも、その考えはコードに埋め込まれ、明文化されません。
設計書として明文化されていないならば、実装に不具合があって設計書に戻ったとしても、検討すべき情報がそこにはないということになります
かくして、設計と実装の乖離ができあがるというわけです。

NOTE: 「モデル駆動開発（MDD）」という開発手法に、「モデル・タイプマッピング」と呼ばれる、2つのモデルの間の対応づけについての考え方があります。モデルとコードの対応づけは、タイプマッピングの一種とみなすこともできます。

==== 状態・イベント・状態遷移・アクション

ステートマシン図で処理を表す時に登場する要素について、簡単に整理しておきましょう。

===== 状態とイベント

「イベント」は、そのクラス（やメソッド）が、そのことが起きないと処理を先に進められなくなるようなできごとのことです。
たとえば、外部からの入力（操作待ちや受信待ちなど）や、時間経過などがイベントの候補になります。
「状態」は、イベントの発生を待っているところです。

===== 状態遷移

状態遷移は、ある状態において待っていたイベントが発生して、別の状態へ移ることを表します。
遷移先が2つ以上ある状態では、各々の状態遷移で待っているイベントは異なっている必要があります。
それは、同じイベントで別の状態へ遷移する状態遷移が複数あると、どちらへ遷移すべきかをイベントの違いだけでは判定できなくなるからです。そのような状態遷移を「あいまいな状態遷移」と呼びます。
「選択疑似状態」を使うと、イベントの発生時に状況によって遷移先を振り分けることもできます。

状態遷移には、イベントが発生したとき、実際に状態を遷移するか判定する条件を追加できます。
この条件のことを「ガード条件」と呼びます。
ガード条件がある状態遷移では、イベントが発生したときにガード条件を評価します。
ガード条件の評価が真のときは、状態が遷移します。
ガード条件の評価が偽のときは、状態は遷移しませんが、その場合もイベントは起きたとことになります（「イベントが消費された」といいます）。

===== アクション

状態や状態遷移には、イベントが発生したときに実行したい処理を割り当てられます。
この処理のことを「アクション（またはエフェクト）」と呼びます。

状態遷移に割り当てられたアクションの場合、イベントが発生すると状態遷移に伴うアクションが実行されます。
状態が遷移するのは、アクションの実行が完了してからです。
ガード条件つきの状態遷移の場合は、イベントの発生後さらにガード条件も真なら、アクションが実行され、次の状態へ遷移します。

状態にアクションを割り当てる場合、次の3つの内部処理を記載できます。

.状態内のアクション（内部動作）の種類
entry / 入場時アクション:: その状態に入った直後に一度だけ実行される処理。
do / Doアクティビティ:: 状態に滞在している間、継続的に（または繰り返し）実行される処理。
exit / 退場時アクション:: イベントが発生してその状態から遷移して抜けるとき、その直前に実行される処理。

==== 状態遷移を担当するメソッドを作成する

ステートマシン図に書いた状態遷移を担当する操作「play」を `play` メソッドとして作成します（ <<ruby_sample03>> ）。
ルールの検討用なので、処理の中に動作確認用の表示処理を書いておきます。

[[ruby_sample03]]
.【Ruby】状態遷移を担当するメソッドを作成する
[source,ruby]
----
class Sample
  def initialize
    # 略
  end

  def play(evt, param)
    puts "#{@state} ->" # <1>
    puts "  event:#{evt}, param: #{param}" # <2>
    case @state # <3>
    when :ST0
      st0_proc(evt, param) # <4>
    when :ST1
      st1_proc(evt, param)
    when :ST2
      # none
    end
    puts "       -> #{@state}" # <5>
    puts 'finished.' if @state == :ST2 # <6>
  end
end
----
<1> 遷移前の状態を表示する。
<2> イベントとパラメーターを表示する。
<3> 状態ごとの処理を case文を使って分岐する。
<4> 状態ごとの詳細な処理を担当するメソッドを呼び出す（ここでは `st0_proc` ） 。
<5> 遷移後の状態を表示する。
<6> 終了状態になったことを知らせる。

==== ガード条件の判定用メソッドを作成する

このサンプルのステートマシン図には、状態遷移のガード条件として「gd1」「gd2」が登場します。
これらをRubyのメソッドとして作成します。
Rubyでは、真偽値を返すメソッドには「 `?` 」をつけるのが慣習です。
この慣習に従って、メソッド名は「 `gd1?` 」「 `gd2?` 」としましょう。
これらのメソッドには、動作確認用に属性の値を使った演算を割当てました（ <<ruby_sample04>> ）。

[[ruby_sample04]]
.【Ruby】ガード条件のメソッドを作成する
[source,ruby]
----
class Sample
  def initialize
    # 略
  end

  def play
    # 略
  end

  def gd1? # <1>
    @attr_a && @attr_b # <2>
  end

  def gd2? # <3>
    !@attr_a || @attr_b
  end
end
----
<1> 「gd1」用のメソッド。真偽値を返すメソッドに「?」をつけるRubyの習慣に倣った。
<2> 属性値を使った条件式の例。
<3> 「gd2」用のメソッド。


==== 状態遷移を追加する（1）

状態ごとの処理を受け持つメソッドを作成します。
メソッド名は、状態名を小文字して後ろに「_proc」をつけるというルールにします。
このルールに従うと、状態「ST0」の場合、メソッド名は「st0_proc」になります（ <<ruby_sample05>> ）。

[[ruby_sample05]]
.【Ruby】「ST0」の状態遷移のメソッドを作成する
[source,ruby]
----
class Sample
  def initialize
    # 略
  end

  def play
    # 略
  end

  def st0_proc(evt, param)
    case evt # <1>
    when :ev1 # <2>
      if gd1? # <3>
        puts "    gd1: #{gd1?}"
        act1(evt, param) # <4>
        @state = :ST1 # <5>
      else # <6>
        puts "    <<< gd1: #{gd1?}, transition is ignored. >>>"
      end
    when :ev3 # <7>
      act3(evt, param)
      @state = :ST2
    end
  end

  # gd1?, gd2? が続く
end
----
<1> イベントで場合分けする。
<2> イベントが「ev1」の場合。イベントはRubyのシンボルを使って表す。
<3> ガード条件による判定。
<4> アクション「act1」の呼び出し。
<5> 次の状態「ST1」への遷移。
<6> ガード条件が偽だったとき。イベントが無視されたことを表示する。
<7> イベントが「ev3」の場合。

==== 状態遷移を追加する（2）

こんどは、状態「ST1」からの状態遷移に対するメソッド「st1_proc」です（ <<ruby_sample07>> ）。

[[ruby_sample07]]
.【Ruby】「ST1」の状態遷移のメソッドを作成する
[source,ruby]
----
class Sample
  def initialize
    # 略
  end

  def play
    # 略
  end

  def st0_proc(evt, param)
    # 略
  end

  def st1_proc(evt, param)
    case evt
    when :ev2 # <1>
      act2(evt, param) # <2>
      puts "    gd2: #{gd2?}"
      if gd2? # <3>
        @state = :ST2 # <4>
      else
        act3(evt, param) # <5>
        @state = :ST0 # <6>
      end
    when :ev3 # <7>
      act3(evt, param)
      @state = :ST0
    end
  end

  # gd1?, gd2? が続く
end
----
<1> イベントが「ev2」の場合。
<2> アクション「act2」の呼び出し。
<3> 選択疑似状態におけるガード条件による判定。
<4> 次の状態「ST0」への遷移。
<5> 選択疑似状態からの遷移におけるアクション「act3」の呼び出し。
<6> 次の状態「ST2」への遷移。
<7> イベントが「ev3」の場合。

[sidebar]
.選択疑似状態がある状態遷移をRubyのコードで表すルール
--
. 選択疑似状態の前の状態遷移については、<<ruby_sample06>> のルールで賄う。

このサンプルでは、アクション「act1」、「act2」、「act3」は、定めたルールを確認しやすいメソッドにしておきます（ <<ruby_sample09>> ）。

さて、これらのアクションと、先に挙げたガード条件をチェックするメソッドを使うのは、「Sample」クラス内部のメソッドだけです。
そこで、Rubyのコードにおいても、これらのメソッドを非公開にしましょう。
Rubyでは、「private」キーワードを追加してその後に非公開のメソッドを配置します。

[[ruby_sample09]]
.【Ruby】「ST1」の状態遷移のメソッドを作成する
[source,ruby]
----
class Sample
  # initialize、play、st0_proc、st1_proc の定義を省略

  praivate # <1>

  def act1(evt, prm)
    puts "     act1: event:#{evt}, param: #{prm}"
  end

  def act2(evt, prm)
    puts "     act2: event:#{evt}, param: #{prm}"
  end

  def act3(evt, prm)
    puts "     act3: event:#{evt}, param: #{prm}"
  end

  def gd1? # <2>
    @attr_a && @attr_b
  end

  def gd2?
    !@attr_a || @attr_b
  end
end
----
<1> この宣言以降のメソッドは、可視性が「private」なメソッドになる。
<2> ガード条件の判定用メソッドも「private」なメソッドに含めた。

==== 構造のモデルだけではプログラムは作れない

構造のモデルを作ったことで、プログラムを構成する要素と、要素同士の関係（関連）を表せるようになりました。
また、クラスのインスタンスをオブジェクト図に見合うように作成すれば、実際に観察したスコアシートと同じようなデータ構造を作成できることもわかりました。

ところで、作りたかったのは、ボウリングのゲームスコアをつけるプログラムでした。
つまり、プレーヤーがフレームごとに交代して投球し、そのたびにスコアを更新する処理です。
ですが、これまでに作成した構造のモデルでは、ゲームの進行に応じてどんなことを処理するか表せていません。
そのためには、処理の中身を表す「振る舞いのモデル」を作成する必要があります。
次の章では、ステートマシン図を使って、フレームやスコアの振る舞いを表すモデルを作成します。
